# MDVWB: веб-интерфейс и взаимодействие с фанкойлами

> Документ предназначен для пользователя, наладчика и инженера эксплуатации.  
> Он описывает статическую страницу настройки MDVWB, управление процессами шин, поиск устройств и взаимодействие с отдельными фанкойлами.

## 1. Назначение веб-интерфейса

Страница MDVWB используется для:

- просмотра текущей конфигурации RS-485-шин;
- добавления новых шин;
- изменения serial port;
- изменения списка адресов;
- включения и отключения шин;
- сохранения общей конфигурации;
- запуска, остановки и перезапуска отдельного процесса;
- просмотра состояния каждого процесса;
- запуска поиска устройств на выбранной шине;
- просмотра найденных адресов.

Страница **не является полноценной панелью оперативного управления всеми фанкойлами**.

Управление отдельными фанкойлами выполняется:

- через устройства, созданные MDVWB в Wiren Board;
- через MQTT-топики `/devices/Fan-...`;
- через другое прикладное веб-приложение, которое использует этот MQTT-контракт.

## 2. Адрес страницы

Стандартный адрес:

```text
http://<IP-адрес-WB>/mdvwb/
```

Пример IPv4:

```text
http://192.168.42.1/mdvwb/
```

Пример IPv6:

```text
http://[2a02:6bf:8013:c01:8f8e:3872:b65c:af82]/mdvwb/
```

Завершающий `/` рекомендуется использовать всегда.

## 3. Расположение файлов

На Wiren Board страница находится в:

```text
/var/www/mdvwb
```

В каталоге должны быть:

```text
index.html
app.js
model.js
mqtt-client.js
styles.css
```

Проверка:

```bash
ls -la /var/www/mdvwb
```

Штатную конфигурацию веб-сервера Wiren Board менять не требуется.

## 4. Требования к браузеру

Рекомендуется современный браузер на базе Chromium:

- Chrome;
- Chromium;
- Яндекс Браузер;
- Microsoft Edge.

Страница:

- не требует интернет;
- не загружает внешние библиотеки;
- работает как набор статических файлов;
- подключается к MQTT через WebSocket `/mqtt`;
- автоматически использует host открытой страницы;
- автоматически пытается восстановить соединение после обрыва.

## 5. Первый запуск

После открытия страницы:

1. Загружается HTML и JavaScript.
2. Браузер подключается к MQTT WebSocket `/mqtt`.
3. Страница подписывается на топики менеджера.
4. Получает retained-конфигурацию `/mdvwb/config`.
5. Создаёт карточки всех шин.
6. Получает retained-статусы процессов.
7. Получает последний retained-результат discovery.
8. Отображает готовый интерфейс.

Если конфигурация ещё не получена, страница показывает ожидание данных.

## 6. Индикатор MQTT

В верхней части страницы отображается состояние соединения.

Типовые состояния:

```text
Подключение…
Подключено
Переподключение…
Отключено
Ошибка MQTT
```

При отключённом MQTT:

- текущие retained-данные могут оставаться видимыми;
- редактирование черновика возможно;
- сохранение конфигурации недоступно;
- кнопки управления процессами недоступны;
- discovery недоступен.

## 7. Сводные показатели

Страница показывает основные числа:

```text
Настроено шин
Активных в конфигурации
Работающих процессов
```

Значения означают:

### Настроено шин

Количество объектов в:

```text
/etc/mdvwb/buses.json
```

### Активных в конфигурации

Количество шин с:

```json
"enabled": true
```

### Работающих процессов

Количество экземпляров:

```text
mdvwb@N.service
```

которые systemd считает активными.

Активная в конфигурации шина может временно быть остановлена вручную. Поэтому второе и третье значения могут отличаться.

## 8. Карточка шины

Для каждой шины отображается отдельная карточка.

Она содержит:

- номер шины;
- признак `enabled`;
- serial port;
- список адресов;
- состояние systemd-процесса;
- состояние автозапуска;
- результат последней команды;
- состояние discovery;
- найденные discovery-адреса;
- кнопки управления.

Пример:

```text
Шина 2
Активна
/dev/ttyUSB0
Адреса: 1, 5, 18
Процесс: работает
Автозапуск: включён
```

## 9. Номер шины

Номер шины:

- находится в диапазоне `1..999`;
- уникален;
- является частью MQTT-имени устройств;
- является номером systemd instance;
- не обязан совпадать с физическим номером порта;
- не обязан идти подряд.

Пример:

```json
{
  "id": 4,
  "port": "/dev/ttyUSB0"
}
```

создаёт:

```text
mdvwb@4.service
Fan-4_1
Fan-4_2
sist-4
```

Допустима конфигурация с шинами:

```text
1, 2, 4, 10
```

## 10. Serial port

Поле порта содержит путь Linux:

```text
/dev/ttyRS485-1
/dev/ttyRS485-2
/dev/ttyUSB0
/dev/serial/by-id/...
```

Правила:

- путь начинается с `/dev/`;
- один port используется только одной шиной;
- port должен существовать на контроллере;
- два процесса не должны открывать один port;
- для USB рекомендуется стабильный `/dev/serial/by-id/...`, если доступен.

Проверка доступных портов:

```bash
ls -l /dev/ttyRS485-* /dev/ttyUSB* /dev/ttyACM* 2>/dev/null
```

Стабильные USB-ссылки:

```bash
find /dev/serial/by-id -maxdepth 1 -type l -ls 2>/dev/null
```

## 11. Адреса фанкойлов

На каждой физической шине адреса находятся в диапазоне:

```text
0..63
```

Пример ввода:

```text
1, 2, 3, 18
```

Страница:

- удаляет пробелы;
- преобразует значения в числа;
- сортирует их;
- отклоняет повторения;
- отклоняет значения вне диапазона.

Строка:

```text
18, 3, 1, 2
```

будет сохранена как:

```text
1, 2, 3, 18
```

Один адрес может существовать на разных физических шинах.

Например:

```text
Fan-1_1
Fan-2_1
```

являются разными устройствами.

## 12. Активная и отключённая шина

### Активная шина

```json
"enabled": true
```

Требования:

- должен быть указан port;
- должен быть хотя бы один адрес;
- после применения менеджер обеспечивает запуск сервиса.

### Отключённая шина

```json
"enabled": false
```

Поведение:

- сервис должен быть остановлен;
- автозапуск должен быть отключён;
- шина остаётся в конфигурации;
- port и адреса можно сохранить для будущего включения;
- пустой список адресов разрешён.

Отключение удобно при временном ремонте или отсутствии оборудования.

## 13. Черновик конфигурации

Редактор использует локальный черновик.

Это означает:

- изменение поля не сразу отправляется на контроллер;
- процессы не перезапускаются при каждом нажатии клавиши;
- можно изменить несколько шин;
- все изменения сохраняются одной операцией.

После изменения появляется сообщение:

```text
Есть несохранённые изменения
```

До сохранения кнопки управления процессами блокируются.

Это предотвращает ситуацию, когда пользователь изменил номер или port только в браузере, а команда отправилась в старую конфигурацию.

## 14. Добавление новой шины

Нажмите:

```text
Добавить шину
```

Заполните:

```text
Номер
Порт
Адреса
Активна
```

Пример:

```text
Номер: 3
Порт: /dev/ttyUSB1
Адреса: 1, 5, 18
Активна: да
```

После добавления запись находится только в черновике.

Нажмите:

```text
Сохранить конфигурацию
```

Менеджер должен:

1. проверить JSON;
2. сохранить `/etc/mdvwb/buses.json`;
3. создать `/etc/default/mdvwb-3`;
4. включить и запустить `mdvwb@3.service`;
5. опубликовать статус;
6. обновить страницу.

## 15. Изменение существующей шины

Можно изменить:

- номер;
- port;
- список адресов;
- enabled.

После сохранения активная изменённая шина перезапускается.

Остальные неизменённые шины продолжают работать.

Пример:

```text
Шина 2:
старый port: /dev/ttyUSB0
новый port: /dev/ttyUSB1
```

После сохранения перезапускается только:

```text
mdvwb@2.service
```

## 16. Удаление шины

Нажмите кнопку удаления в карточке или редакторе.

До общего сохранения удаление существует только в черновике.

После сохранения менеджер:

- останавливает экземпляр;
- отключает его автозапуск;
- удаляет управляемый `/etc/default/mdvwb-N`;
- очищает retained-топики удалённой шины и её устройств;
- оставляет остальные процессы без изменений.

Удаление записи не удаляет физическое оборудование.

## 17. Отмена изменений

Кнопка:

```text
Отменить изменения
```

возвращает черновик к последней конфигурации, полученной из:

```text
/mdvwb/config
```

Команда не изменяет файлы на контроллере.

## 18. Сохранение конфигурации

Кнопка:

```text
Сохранить конфигурацию
```

публикует JSON в:

```text
/mdvwb/config/set
```

Команда отправляется с:

```text
retain=false
```

Результат приходит в:

```text
/mdvwb/config/result
```

### Успешный результат

```json
{
  "success": true,
  "saved": true,
  "message": "Configuration saved and applied"
}
```

### Ошибка валидации

```json
{
  "success": false,
  "saved": false,
  "message": "..."
}
```

### Файл сохранён, но systemd не применился

```json
{
  "success": false,
  "saved": true,
  "message": "Configuration saved, service synchronization failed: ..."
}
```

В последнем случае новая конфигурация уже находится на диске. Следует проверить менеджер и systemd.

## 19. Ошибки редактора

### Повторяющийся номер шины

Пример ошибки:

```text
Шина 2 указана несколько раз
```

### Повторяющийся port

```text
Порт /dev/ttyUSB0 назначен нескольким шинам
```

### Неправильный port

```text
Порт должен начинаться с /dev/
```

### Повторяющийся адрес

```text
Адрес 5 повторяется
```

### Адрес вне диапазона

```text
Адрес должен быть от 0 до 63
```

### Активная шина без адресов

```text
Для активной шины нужно указать хотя бы один адрес
```

Пока есть ошибки, сохранение недоступно.

## 20. Кнопки управления процессом

В карточке шины доступны:

```text
Запустить
Остановить
Перезапустить
Обновить
Найти устройства
```

Кнопки работают через MQTT и `mdvwb-manager`.

## 21. Запуск шины

Кнопка:

```text
Запустить
```

отправляет:

```text
/mdvwb/buses/<id>/start
```

Условия:

- MQTT подключён;
- нет несохранённого черновика;
- шина существует в config;
- `enabled=true`;
- discovery не выполняется.

Ручной запуск не изменяет `buses.json`.

## 22. Остановка шины

Кнопка:

```text
Остановить
```

отправляет:

```text
/mdvwb/buses/<id>/stop
```

Ручная остановка:

- останавливает процесс;
- не изменяет `enabled`;
- не изменяет автозапуск;
- не удаляет config;
- может быть отменена следующим `apply` или restart менеджера.

Для долговременного отключения измените:

```json
"enabled": false
```

и сохраните config.

## 23. Перезапуск шины

Кнопка:

```text
Перезапустить
```

отправляет:

```text
/mdvwb/buses/<id>/restart
```

Перезапускается только выбранный процесс.

Перезапуск полезен:

- после восстановления физического порта;
- после временной ошибки serial;
- после изменения внешнего оборудования;
- для повторной публикации metadata и начального состояния.

## 24. Обновление статуса

Кнопка:

```text
Обновить
```

отправляет:

```text
/mdvwb/buses/<id>/status/get
```

Менеджер публикует новый retained-статус:

```text
/mdvwb/buses/<id>/status
```

Пример:

```json
{
  "id": 2,
  "configured": true,
  "enabled": true,
  "service": "active",
  "autostart": true,
  "port": "/dev/ttyUSB0",
  "addresses": [1, 5, 18]
}
```

## 25. Состояния процесса

Типовые значения:

```text
Работает
Остановлена
Ошибка
Неизвестно
```

### Работает

systemd считает `mdvwb@N.service` активным.

Это не гарантирует, что каждый фанкойл отвечает. Состояние отдельных устройств смотрите через `Alarm` и `Status`.

### Остановлена

Процесс не выполняет опрос.

### Ошибка

Процесс завершился с ошибкой или systemd не смог его запустить.

Смотрите:

```bash
journalctl -u mdvwb@N.service -n 100 --no-pager
```

## 26. Запуск discovery

Кнопка:

```text
Найти устройства
```

отправляет:

```text
/mdvwb/buses/<id>/discovery/start
```

Перед запуском показывается предупреждение:

```text
Сервис этой шины будет остановлен и после поиска автоматически не запустится
```

Это ожидаемое поведение.

## 27. Что делает discovery

Менеджер:

1. Находит выбранную шину в `buses.json`.
2. Берёт её port.
3. Останавливает только `mdvwb@N.service`.
4. Запускает `MDVWB --discover`.
5. Сканирует `0..63`.
6. Выполняет три полных прохода.
7. Публикует найденные адреса.
8. Оставляет сервис остановленным.

Остальные шины продолжают работать.

## 28. Результат discovery

Статус:

```text
/mdvwb/buses/<id>/discovery/status
```

Результат:

```text
/mdvwb/buses/<id>/discovery/result
```

Пример:

```json
{
  "success": true,
  "bus": 3,
  "addresses": [1, 5, 18],
  "message": "Discovery completed"
}
```

Результат retained, поэтому отображается после повторного открытия страницы.

## 29. Найденные адреса не применяются автоматически

Это обязательное правило безопасности.

Discovery:

- не изменяет `buses.json`;
- не добавляет адреса;
- не удаляет старые адреса;
- не запускает сервис.

После поиска пользователь должен:

1. Проверить найденный список.
2. Открыть редактор шины.
3. Вручную изменить addresses.
4. Сохранить config.
5. Запустить шину, если она не запустилась в результате применения.

## 30. Почему discovery оставляет шину остановленной

Автоматический запуск может быть опасен:

- найденный список ещё не проверен;
- в конфигурации могут оставаться неправильные адреса;
- пользователь может выполнять монтаж;
- port может использоваться диагностическим инструментом;
- повторный опрос может мешать дальнейшим тестам.

Явный запуск делает поведение предсказуемым.

## 31. Demonstration mode

Локальный режим:

```text
http://localhost:8080/mdvwb/?demo=1
```

Он используется для проверки интерфейса без Wiren Board.

В demo mode:

- данные искусственные;
- команды не отправляются в реальный MQTT;
- конфигурация не сохраняется на контроллер;
- результаты процессов и discovery имитируются.

Не используйте demo mode для оценки реального оборудования.

## 32. Устройства фанкойлов в Wiren Board

Драйвер публикует retained metadata и создаёт устройства:

```text
Fan-<bus>_<address>
```

Примеры:

```text
Fan-1_1
Fan-1_18
Fan-2_1
```

Также создаётся системное устройство шины:

```text
sist-<bus>
```

Примеры:

```text
sist-1
sist-2
```

Старый wb-rules-массив `ArrID` для создания этих устройств не нужен.

## 33. Где смотреть фанкойлы

В стандартном интерфейсе Wiren Board найдите устройство по имени:

```text
Fan-1_1
```

Оно отображает состояния, опубликованные драйвером.

Точное расположение устройства в интерфейсе зависит от версии стандартного web UI Wiren Board.

## 34. Ограничение текущей metadata

В текущей версии metadata контролов публикуется как readonly.

Это означает:

- стандартный интерфейс WB надёжно показывает фактические значения;
- возможность редактирования конкретного control штатным UI может отсутствовать;
- основной стабильный способ отправки команд — MQTT topic `/on1`;
- отдельные прикладные интерфейсы должны использовать MQTT-контракт MDVWB.

Изменять readonly metadata без проверки стандартного интерфейса Wiren Board не следует.

## 35. MQTT-имя фанкойла

Формат:

```text
Fan-<bus>_<address>
```

Пример:

```text
Fan-2_18
```

Соответствующий base:

```text
/devices/Fan-2_18
```

## 36. Фактические контролы

```text
Power
Mode
Speed
SetTemp
Temp
Blinds
Blok
Alarm
AlarmCode
Status
```

Топик:

```text
/devices/Fan-<bus>_<address>/controls/<Control>
```

Пример:

```text
/devices/Fan-1_3/controls/Temp
```

Фактические значения:

- retained;
- публикуются только после корректного C0;
- публикуются при изменении;
- после запуска доступны новым подписчикам сразу;
- не находятся в `/on1`.

## 37. Командные топики

Формат:

```text
/devices/Fan-<bus>_<address>/controls/<Control>/on1
```

Пример:

```text
/devices/Fan-1_3/controls/Power/on1
```

Правила:

- отправлять без retain;
- payload — одно целое значение;
- topic должен заканчиваться на `/on1`;
- `/on` не используется;
- base control topic является фактическим состоянием, а не командой.

## 38. Power

Фактический topic:

```text
/devices/Fan-1_3/controls/Power
```

Команда:

```text
/devices/Fan-1_3/controls/Power/on1
```

Значения:

```text
0 = выключить
1 = включить
```

Пример включения:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Power/on1' \
  -m '1'
```

Пример выключения:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Power/on1' \
  -m '0'
```

Power независим от Mode. Изменение Power не должно сбрасывать выбранный режим.

## 39. Mode

Команда:

```text
/devices/Fan-1_3/controls/Mode/on1
```

Значения:

| Значение | Режим |
|---:|---|
| `0` | Cool |
| `1` | Heat |
| `2` | Dry |
| `3` | Fan |
| `4` | Auto |

Пример охлаждения:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Mode/on1' \
  -m '0'
```

Пример автоматического режима:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Mode/on1' \
  -m '4'
```

Изменение Mode сохраняет Power.

## 40. Speed

Команда:

```text
/devices/Fan-1_3/controls/Speed/on1
```

Значения:

| Значение | Скорость |
|---:|---|
| `1` | Low |
| `2` | Medium |
| `3` | High |
| `4` | Auto |

Пример:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Speed/on1' \
  -m '2'
```

Значение `0` не является штатной командой скорости.

## 41. SetTemp

Команда:

```text
/devices/Fan-1_3/controls/SetTemp/on1
```

Диапазон драйвера:

```text

```

Пример:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/SetTemp/on1' \
  -m '24'
```

Значение должно быть целым.

После команды фактический `SetTemp` может измениться не сразу.

## 42. Temp

Фактический topic:

```text
/devices/Fan-1_3/controls/Temp
```

Это текущая комнатная температура T1, полученная из C0.

Преобразование внутри драйвера:

```text
raw / 2 - 20
```

Если устройство сообщает `0xFF`, температура считается недоступной.

`Temp` не является командным параметром.

/devices/Fan-1_3/controls/Blok/on1

Значения:

mosquitto_pub \

mosquitto_pub \

## 45. Alarm

Фактический topic:

```text
/devices/Fan-1_3/controls/Alarm
```

Значения:

| Значение | Смысл |
|---:|---|
| `0` | Нормальная работа |
| `1` | Авария фанкойла |
| `2` | Нет связи / offline |

При `Alarm=2` проверьте:

- питание фанкойла;
- RS-485 A/B;
- адрес;
- port;
- журнал сервиса;
- наличие других устройств на линии.

## 46. AlarmCode

Фактический topic:

```text
/devices/Fan-1_3/controls/AlarmCode
```

Значения:

```text
0 = код отсутствует
1..16 = первая обнаруженная ошибка E0..EF
```

Текущая версия публикует числовой код без полного пользовательского справочника расшифровки.

Для инженерной диагностики дополнительно смотрите физическую индикацию фанкойла и документацию конкретной модели.

## 47. Status

Фактический topic:

```text
/devices/Fan-1_3/controls/Status
```

Значения:

| Значение | Смысл |
|---:|---|
| `0` | Выключен |
| `1` | Охлаждение |
| `2` | Нагрев |
| `3` | Осушение |
| `4` | Вентиляция |
| `5` | Auto |
| `6` | Авария |
| `7` | Нет связи |

`Status` рассчитывается по фактическому C0-состоянию.

`Blok` не переводит устройство в отдельный общий статус.

## 48. Подписка на один фанкойл

Все топики:

```bash
mosquitto_sub -v -t '/devices/Fan-1_3/#'
```

Только фактические контролы:

```bash
mosquitto_sub -v \
  -t '/devices/Fan-1_3/controls/Power' \
  -t '/devices/Fan-1_3/controls/Mode' \
  -t '/devices/Fan-1_3/controls/Speed' \
  -t '/devices/Fan-1_3/controls/SetTemp' \
  -t '/devices/Fan-1_3/controls/Temp' \
  -t '/devices/Fan-1_3/controls/Alarm' \
  -t '/devices/Fan-1_3/controls/AlarmCode' \
  -t '/devices/Fan-1_3/controls/Status'
```

## 49. Подписка на все фанкойлы одной шины

```bash
mosquitto_sub -v -t '/devices/Fan-1_+/controls/#'
```

Поддержка `+` зависит от позиции MQTT wildcard.

Надёжный вариант:

```bash
mosquitto_sub -v -t '/devices/+/controls/#'
```

и фильтрация строк `Fan-1_`.

## 50. Подписка на конкретный параметр всех устройств

```bash
mosquitto_sub -v -t '/devices/+/controls/Status'
```

Температура:

```bash
mosquitto_sub -v -t '/devices/+/controls/Temp'
```

Аварии:

```bash
mosquitto_sub -v \
  -t '/devices/+/controls/Alarm' \
  -t '/devices/+/controls/AlarmCode'
```

## 51. Почему фактическое значение может обновиться с задержкой

После команды драйвер выполняет:

1. формирование полного C3;
2. отправку команды;
3. проверку ответа;
4. постановку подтверждающего C0;
5. ожидание фактического состояния;
6. публикацию подтверждённого значения.

Некоторые фанкойлы после C3 возвращают старое значение.

Это нормальное известное поведение.

Не используйте:

```text
.../on1
```

для чтения фактического результата.

Подписывайтесь на:

```text
.../controls/<Control>
```

## 52. Retained-состояния

Фактические состояния и metadata публикуются retained.

Преимущества:

- веб-интерфейс сразу получает последнее состояние;
- Wiren Board сразу восстанавливает карточку устройства;
- подписчику не надо ждать следующего изменения.

Проверка:

```bash
mosquitto_sub -v -C 1 -W 5 \
  -t '/devices/Fan-1_3/controls/Power'
```

Если состояние уже известно, сообщение приходит сразу.

## 53. Retained-команды запрещены

Неправильно:

```bash
mosquitto_pub -r \
  -t '/devices/Fan-1_3/controls/Power/on1' \
  -m '1'
```

Правильно:

```bash
mosquitto_pub \
  -t '/devices/Fan-1_3/controls/Power/on1' \
  -m '1'
```

Драйвер отклоняет retained-команды, чтобы старая команда не выполнилась повторно после перезапуска или переподключения.

## 54. Неизвестное устройство

Команда отклоняется, если:

- адрес отсутствует в config;
- команда относится к другой шине;
- устройство ещё не ответило на C0;
- сервис шины остановлен;
- MQTT topic сформирован неправильно.

Проверьте:

```bash
mdvwb-manager summary /etc/mdvwb/buses.json
systemctl status mdvwb@1.service --no-pager
mosquitto_sub -v -t '/devices/Fan-1_3/#'
```

## 55. Системное устройство шины

Имя:

```text
sist-<bus>
```

Пример:

```text
sist-1
```

Контролы:

```text
Serial
Error
GanGetID
```

Топики:

```text
/devices/sist-1/controls/Serial
/devices/sist-1/controls/Error
/devices/sist-1/controls/GanGetID
```

### Serial

Состояние serial transport.

### Error

Системная ошибка шины или драйвера.

Обычный timeout одного фанкойла отражается в его `Alarm=2` и `Status=7`, а не обязательно в системном `Error`.

### GanGetID

Текущий опрашиваемый адрес.

По умолчанию отключён для снижения MQTT-трафика.

Включается опцией драйвера:

```text
--publish-poll-address
```

## 56. Типовой сценарий ввода новой шины

1. Подключить USB–RS-485.
2. Найти port:

```bash
ls -l /dev/ttyUSB*
```

3. Открыть `/mdvwb/`.
4. Добавить шину с уникальным ID.
5. Указать port.
6. Сначала можно оставить шину отключённой.
7. Сохранить config.
8. Запустить discovery.
9. Проверить найденные адреса.
10. Вручную указать addresses.
11. Включить шину.
12. Сохранить config.
13. Проверить процесс.
14. Проверить `Fan-<bus>_<address>`.
15. Проверить `Alarm` и `Status`.

## 57. Типовой сценарий замены USB-адаптера

1. Остановить шину.
2. Подключить новый адаптер.
3. Проверить новый port.
4. Изменить port в редакторе.
5. Сохранить config.
6. Проверить новый `mdvwb@N.service`.
7. Проверить получение C0.
8. Проверить retained-состояния.

Остальные шины продолжают работать.

## 58. Типовой сценарий добавления фанкойла на существующую шину

1. Выполнить монтаж.
2. Убедиться в уникальности адреса.
3. Остановить выбранную шину.
4. Запустить discovery.
5. Найти новый адрес.
6. Добавить его в addresses.
7. Сохранить config.
8. Запустить шину.
9. Проверить новое устройство.
10. Проверить команды.

## 59. Типовой сценарий временного ремонта

1. В редакторе установить:

```json
"enabled": false
```

2. Сохранить config.
3. Выполнить ремонт.
4. При необходимости запустить discovery.
5. Восстановить addresses.
6. Установить `enabled=true`.
7. Сохранить config.

## 60. Групповое управление

Текущий веб-интерфейс `/mdvwb/` не выполняет групповые команды фанкойлам.

Для группового управления требуется отдельное приложение или wb-rules-логика, которая публикует отдельные `/on1` команды каждому выбранному устройству.

Правила группового управления:

- не использовать broadcast `0xFF`;
- отправлять индивидуальную команду каждому адресу;
- учитывать фактический bus ID;
- каждую команду отправлять non-retained;
- не считать публикацию MQTT подтверждением;
- ждать фактические base topics.

## 61. Расписание

Текущий MDVWB не хранит пользовательское расписание фанкойлов.

Расписание должно реализовываться внешним компонентом:

- wb-rules;
- отдельным веб-приложением;
- системой автоматизации;
- MQTT-сценарием.

Драйвер отвечает только за доставку индивидуальных команд и публикацию фактического состояния.

## 62. Безопасное массовое управление

При управлении множеством устройств:

1. Использовать список конкретных `Fan-<bus>_<address>`.
2. Не использовать broadcast.
3. Публиковать команды последовательно или ограниченными пакетами.
4. Не ставить retain.
5. Проверять `Alarm` и `Status`.
6. Учитывать, что опрос идёт с общим периодом 150 мс на транзакцию каждой шины.
7. Не ожидать мгновенного обновления десятков устройств одновременно.

Разные шины работают параллельно отдельными процессами.

Проверьте:

- питание;

Если адрес указан в config, но устройство не существует:

```text

```

Остальные адреса продолжают опрашиваться.

```text
1
```

Проверьте отсутствие retain.

Проверьте журнал:

## 70. Диагностика дублирующихся устройств

Не должны одновременно работать:

- новый C++ metadata publisher;
- старый wb-rules `ArrID`;
- другой скрипт с `defineVirtualDevice("Fan-")`.

Проверка:

```bash
grep -R \
  -e 'ArrID' \
  -e 'defineVirtualDevice("Fan-"' \
  /etc/wb-rules 2>/dev/null
```

Установщик сохраняет отключённый файл с суффиксом:

```text
.disabled-mdvwb
```

## 71. Диагностика удалённого устройства

Если удалённое устройство осталось в UI, проверьте retained topics:

```bash
mosquitto_sub -v -C 20 -W 5 -t '/devices/Fan-1_63/#'
```

После удаления manager должен очистить stale metadata и состояния.

Перезапуск активной шины повторно публикует только текущие устройства.

## 72. Рекомендации пользователю

- Сначала сохраните config, затем управляйте процессами.
- Перед discovery учитывайте остановку шины.
- После discovery вручную проверяйте addresses.
- Не используйте retained-команды.
- Не используйте broadcast.
- Не редактируйте производные `/etc/default/mdvwb-N` как основной config.
- Не запускайте два экземпляра на одном port.
- Не считайте C3-ответ фактическим подтверждением.
- При сомнениях смотрите base state topic.
- Перед массовыми изменениями создавайте backup `/etc/mdvwb/buses.json`.
